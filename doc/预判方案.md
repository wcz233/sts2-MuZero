# sts2-MuZero 预判方案评估

## 1. 结论

你的方案能解决一个真实问题：

- 当前策略会把“敌人这回合其实已经能被击杀”与“敌人这回合仍然会出手”混在一起。
- 这会直接导致过度防御、攻击延后、甚至“防御致死”。

但是，如果把你描述的整套逻辑直接做成 **主决策器**，即：

1. 先枚举攻击方案
2. 再算最小防御约束
3. 决定必须打几张防御
4. 最后固定为“先防御、再攻击”

那么它 **明显偏离 MuZero / 强化学习在这里的原始目标**。

原因不是“用了搜索就不算强化学习”，而是：

- 你不是在给 MuZero 提供辅助信息，而是在用一个外部规则系统直接替它做战斗决策。
- 训练出来的策略将越来越像“模仿这个硬编码控制器”，而不是自己学会战斗。
- 一旦规则写漏、写错、或覆盖不到复杂卡组，模型会继承这些偏差，而且很难靠训练自己纠正。

所以结论是：

- **不建议** 把该方案作为主控制逻辑直接接管出牌。
- **建议** 把它降级为“战术预判器 / tactical analyzer”，只作为：
  - 观测特征
  - 搜索先验偏置
  - 局部硬约束
  - reward shaping

这不违背当前 sts2-MuZero 的方向，也更容易逐步验证。

## 2. 原方案中合理的部分

以下思想本身是对的，而且值得保留：

- 如果本回合存在击杀方案，那么被击杀的敌人不应再贡献攻击压力。
- 计算攻击收益时，必须考虑：
  - 我方力量等增伤
  - 单次攻击增伤
  - 敌方易伤
  - 敌方反伤 / 荆棘 / 反击
  - 伤害上限类效果
- 计算防御约束时，应基于“击杀后剩余敌人还能造成多少伤害”，而不是基于当前敌人列表的原始总伤害。
- 目标不应是“堆满格挡”，而应是“在满足生存约束后，尽量把剩余资源转成输出”。

这些都可以保留。

## 3. 原方案中不适合作为主决策器的部分

### 3.1 直接决定动作序列，实质上是在替代策略

MuZero 当前在本仓库中的结构是：

- 网络输出 policy/value
- 根节点搜索只接受少量环境侧 `prior_biases`
- 最终动作仍由 MuZero 搜索分布采样/选择

如果改成：

- 外部规则系统先算出“该打几张防御、该打哪些攻击、先后顺序是什么”
- 再让 MuZero 只在这个结果附近活动

那 MuZero 就不再是主策略，而变成一个“给规则系统打补丁的次级模块”。

这会带来几个后果：

- exploration 被强压缩
- replay 中充满硬编码策略轨迹
- 模型学不到规则系统没覆盖到的例外情况
- 后续调参会越来越像调启发式，而不是调学习系统

### 3.2 “先防御，再攻击”是错误的硬顺序

这个顺序在很多卡组里并不成立。

反例很常见：

- 先攻击可击杀敌人，直接降低 incoming damage，此时再防御需求会下降。
- 某些攻击/技能本身会给格挡、抽牌、降敌伤、上 debuff。
- 某些牌序要求先上力量/易伤/费用变化，再打攻击。
- 某些防御牌会抽牌，先打反而可能开出更优击杀线。

所以“先防御后攻击”不应是硬规则。

正确表达应该是：

- **搜索所有当前回合可执行序列**
- 计算每个序列在“输出 / 生存 / 副作用”上的结果
- 找出最优或近似最优序列

这和“先算防御张数，再固定顺序执行”不是一回事。

### 3.3 Alpha-Beta 并不适合这个问题

Alpha-Beta 适合：

- 双方轮流决策
- 零和
- 对手在搜索层中显式行动

而 STS 的“当前我方回合出牌”更像：

- 单边组合优化
- 当前敌方 intent 已知
- 当前回合内大多是确定性状态转移

所以这里更合适的不是 Alpha-Beta，而是：

- **当前回合确定性战术求解器**
- 形式上更接近：
  - DFS + 剪枝
  - branch and bound
  - memoization / transposition table
  - beam search（如果分支太大）

Monte Carlo 也不是首选，因为你要算的是“当前回合是否存在击杀线 / 最小剩余伤害”，这类问题更需要确定性求解，不需要随机采样。

## 4. 更好的方案：把它做成“辅助预判器”，而不是主控制器

### 4.1 定位

新增一个模块：

- `combat_tactical_solver.py`

它不直接替 MuZero 下决定，而只输出若干战术结果。

### 4.2 只做“当前回合”求解，不做多回合博弈

首版范围建议严格限制为当前回合：

- 输入：
  - 当前手牌
  - 当前能量
  - 敌方当前 intent
  - 当前敌我 buff/debuff
  - 当前 potions / 可选目标
- 输出：
  - `lethal_exists`
  - `best_kill_set`
  - `best_sequence`
  - `min_incoming_after_best_sequence`
  - `min_required_block_after_best_kill`
  - `retaliation_damage_under_best_sequence`
  - `best_damage_total`
  - `best_survival_margin`

注意：

- 这不是“预测未来几回合”
- 只是“精算当前回合所有合理序列的结果”

这样复杂度可控，也最符合你要解决的问题。

### 4.3 求解目标函数

不要写成“先防御再攻击”，而应写成一个统一打分函数。

建议当前回合目标按优先级排序：

1. 生存优先
2. 在生存成立前提下，最小化本回合结束后的有效 incoming damage
3. 在上述成立前提下，最大化击杀数 / 减少危险敌人数
4. 再最大化输出、debuff、资源利用率

可以表达成字典序目标：

1. `survive`
2. `min_post_turn_hp_loss`
3. `max_kill_weighted_threat_removed`
4. `max_damage`
5. `min_energy_waste`

### 4.4 状态转移需要统一，不要按牌名硬写

如果真做求解器，最关键的是：

- 不能只靠“攻击牌就是伤害、防御牌就是格挡”这种粗粒度分类。

至少要统一处理：

- 伤害
- 格挡
- 力量
- 易伤
- 虚弱
- 反伤 / 荆棘 / 反击
- 一次性增伤
- 伤害上限
- 击杀后敌人不再出手
- 攻击后自身受到的即时反伤

否则求解器会给出错误结论。

## 5. 与 MuZero 的正确集成方式

### 5.1 第一层：作为 observation feature

把以下量加入状态特征：

- `lethal_exists`
- `best_kill_count`
- `min_required_block_after_best_kill`
- `min_required_block_after_best_damage`
- `best_retaliation_damage`
- `best_post_turn_incoming_damage`
- `best_survival_margin`
- `best_attack_if_no_block`

这样模型会“看到”真正有意义的战术量，而不是只能从原始 hp/block/incoming 自己慢慢猜。

这是最不违背 MuZero 的方式。

### 5.2 第二层：作为 action prior bias

在 `build_action_prior_biases()` 中使用它，而不是直接接管动作。

例如：

- 如果 `lethal_exists=True`，提升参与最佳击杀线的动作 prior
- 如果 `min_required_block_after_best_kill <= current_block`，降低纯防御牌 prior
- 如果某张攻击牌会触发致命反伤，则降低该动作 prior

这样 MuZero 仍然在决策，只是搜索被引导到更合理的局部。

### 5.3 第三层：仅对“严格支配动作”做硬过滤

只有在动作是**严格支配**时，才建议硬过滤。

例如：

- 当前 `current_block >= min_required_block_after_best_kill`
- 某张牌是纯防御牌
- 且该牌不抽牌、不产能、不叠力量、不触发关键联动

这时可以认为它在当前回合几乎一定是冗余动作，允许隐藏。

但不要把“所有防御牌都按预判结果强制禁掉”。

### 5.4 第四层：作为 reward shaping

可以新增几类局部 shaping：

- 有击杀线却错过致死：负奖励
- 明显过挡：负奖励
- 在满足生存条件下把额外资源转成伤害：正奖励
- 选择能降低敌方总威胁的击杀目标：正奖励

这样训练会更快，但主策略仍由 MuZero 学习。

## 6. 推荐实现路线

### 方案 A：推荐，偏稳健

目标：

- 不改 MuZero 主体
- 新增战术求解器
- 只输出特征 + prior bias + 少量硬过滤

实施顺序：

1. 做当前回合的确定性求解器
2. 先只输出 `lethal_exists` 与 `min_required_block_after_best_kill`
3. 接入 observation features
4. 接入 `build_action_prior_biases()`
5. 最后只对“纯防御且严格冗余”做硬过滤

优点：

- 风险最低
- 便于离线复盘
- 不会把系统改成规则控制器

### 方案 B：可做，但要非常克制

目标：

- 让求解器直接给出“推荐序列”
- 但 MuZero 只把它当作强 prior，不是强制执行

要求：

- 推荐序列只能转成 bias
- 不能转成唯一合法动作

### 方案 C：不推荐

目标：

- 求解器直接决定“先防御几张，再攻击几张”
- MuZero 只负责剩余细节

问题：

- 这已经不是辅助学习，而是在用手写战斗 AI 接管主策略
- 长期会让训练目标变得混乱

## 7. 算法建议

如果决定做“辅助预判器”，建议算法不是 Alpha-Beta，也不是大范围蒙特卡洛，而是：

- **当前回合 DFS + 剪枝 + memoization**

原因：

- 当前回合问题主要是单边组合优化
- 许多状态是确定的
- 可利用大量剪枝

推荐剪枝：

- 相同状态压缩去重
- 能量相同但伤害/格挡更差的状态淘汰
- 同名纯重复牌的对称枝裁剪
- 已经达到 lethal 的分支不再继续扩展无意义动作
- 防御已经足够且后续纯防御无附加收益时，剪掉继续堆防的分支

状态压缩至少包含：

- 剩余手牌 multiset
- 当前能量
- 当前格挡
- 当前力量/关键 buff
- 敌方 hp / vulnerable / thorns / 关键状态
- 已确定的本回合 retaliate/self-damage

## 8. 与当前仓库的落点建议

建议挂接点：

- `sts2_muzero/env.py`
  - `build_action_prior_biases()`
  - `_redundant_defense_card_indexes()`
  - `_combat_tactical_potential()`
  - `_compute_reward()`
- 新增：
  - `sts2_muzero/combat_tactical_solver.py`

建议日志记录：

- `lethal_exists`
- `min_required_block_after_best_kill`
- `best_sequence_summary`
- `best_kill_target`
- `best_retaliation_damage`

这样后面能直接从 logs 里分析：

- 为什么这回合还在防御
- 为什么它判断可以不防
- 为什么它选这个击杀目标

## 9. 最终建议

最终建议很明确：

- **不要** 直接实现成“规则控制器版战斗 AI”。
- **要** 实现成“当前回合战术预判器”，并把结果以：
  - 特征
  - prior
  - 轻量硬约束
  - reward shaping
  的形式接入 MuZero。

一句话概括：

- **可以做预判，但不要让预判替 MuZero 做决定。**

如果你后续决定实施，我建议先做一个最小版本：

1. 只解当前回合
2. 只输出 `lethal_exists`
3. 只输出 `min_required_block_after_best_kill`
4. 先接 observation 和 prior
5. 不立即接“固定先防御再攻击”的硬控制

这是当前最稳、最不容易把系统带歪的路线。
